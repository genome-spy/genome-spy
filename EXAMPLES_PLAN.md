# Examples Reorganization Plan

## Purpose

This document describes how to reorganize GenomeSpy examples so that:

- docs examples become reusable file-based assets instead of inline-only Markdown blocks
- the app dev server can browse and open the same shared examples
- the playground can open the same examples reliably
- example validation can be automated
- local developer-only projects remain ergonomic
- large website showcase examples can stay outside this repository

The plan reflects the current codebase and the additional constraints discussed during design.

## Summary

The repository should have three distinct example sources:

1. `examples/`
   Shared, curated, committed examples in the `genome-spy` repo.
2. `private/`
   Eventual repo-root home for local developer-only projects in the `genome-spy` repo, not committed.
3. `website-examples`
   Large public showcase examples kept outside `genome-spy`, currently mounted into `genome-spy.github.io/examples/` as a submodule.

The shared `examples/` directory becomes the canonical source for:

- docs embeds
- docs code snippets
- app dev-server browsing
- playground loading
- example validation tests

Docs should inline example JSON into the page to avoid layout shifts. A docs macro should still know the originating example file path so it can:

- show the exact file contents
- set the correct docs embed `baseUrl`
- add an `Open in Playground` link

Extracted docs examples should become proper first-class spec files, including:

- a `$schema` entry
- a succinct `description` suitable for later use in autogenerated catalogs

The playground should stop relying on fragile hidden base URL state. It must distinguish between:

- shared monorepo examples loaded against the examples root
- external website examples loaded from their own file directory
- local developer projects under `private/`, which should behave like standalone spec folders

## Goals

- Make docs examples reusable by file path instead of copy-pasted inline JSON.
- Establish a discoverable repo-root home for shared examples.
- Preserve convenient local development for large custom projects with colocated data.
- Support both public shared examples and external website examples without copying large data into this repo.
- Replace the current playground base URL hack with predictable behavior.
- Enable automated validation of the shared example set.
- Keep local playground development and deployed `https://genomespy.app/playground/` as similar as practical for loading curated shared examples.

## Non-goals

- Move the large `website-examples` dataset into this repository.
- Make `private/` part of docs, CI docs assets, or public manifests.
- Build a full Vega-style image rendering pipeline for examples.
- Solve every possible ad hoc external example hosting case in the first step.

## Current State

### Shared examples in the monorepo

- File-based examples currently live under `packages/core/examples/`.
- The app dev server in `packages/app/dev-server.mjs` serves them at `/examples`.
- The app single-page shell points users to `?spec=examples/first.json`.
- Tests such as `packages/core/examples/layout.test.js` reference example specs manually.

### Docs examples

- Interactive docs examples are embedded inline in Markdown using `<genome-spy-doc-embed>`.
- The docs runtime comes from `packages/doc-embed/index.js`.
- There is no current docs macro for including example specs by file path.
- Docs asset staging currently copies built docs runtime assets and schema files into `docs/`.

### Playground

- The playground can load `?spec=...` and derives a `baseUrl` from the URL string.
- The current implementation used to keep or clear the query string in ad hoc ways, which made source tracking inconsistent.
- The current implementation stores only spec text in localStorage.
- This becomes unreliable when a previously URL-backed spec is later restored from localStorage without its source context.

### Website examples

- `genome-spy.github.io` contains an `examples` submodule pointing to `website-examples`.
- Those examples include sizeable data files and shared imported fragments.
- They are currently linked to the deployed playground using relative cross-app URLs such as `playground/?spec=../examples/...`.
- Those examples should remain outside this repository.

### Developer-only private projects

- Local-only work now lives under repo-root `private/`.
- These projects often live in subdirectories containing both spec files and colocated data.
- The app dev server serves repo-root `private/` at `/private` and still supports legacy `packages/core/private/` as a compatibility fallback.
- This workflow is important and must remain easy to use.

## Key Decisions

## 1. Canonical shared examples move to repo-root `examples/`

Rationale:

- shared between docs, dev server, tests, and playground
- more discoverable at the monorepo root
- avoids implying examples belong to the `core` package only

Consequences:

- `packages/core/examples/` becomes a migration source, then either a compatibility shim or removed entirely
- scripts, tests, and docs need to repoint to the new root

Subdirectories should initially include:

- `examples/core/` for examples migrated from `packages/core/examples/`
- `examples/docs/` for examples extracted from embedded Markdown blocks
- `examples/app/` for examples that require app-only functionality

## 2. Developer-only work lives under repo-root `private/`

Rationale:

- consistent with `examples/`
- better reflects actual usage: repo-level local projects, not package internals
- keeps the local sandbox workflow intact

Consequences:

- dev-server route stays `/private`
- repo-root `private/` is the primary home for local developer-only projects
- legacy `packages/core/private/` remains supported as a compatibility fallback
- neither location is part of docs/public examples/test manifests

## 3. `website-examples` stays external

Rationale:

- avoids bringing tens of megabytes of example data into this repo
- preserves existing separation between public showcase examples and shared monorepo examples

Consequences:

- playground must support opening examples from both monorepo-hosted and external hosted paths
- docs/public shared examples should not depend on `website-examples`

## 4. Docs should inline spec text, not fetch spec JSON at runtime

Rationale:

- prevents layout shifts caused by async JavaScript loading after initial page render
- keeps docs output self-contained and predictable

Consequences:

- docs macro must read example files during docs build
- docs embed component must accept an explicit base URL for relative assets

## 5. Shared monorepo examples should be authored against the examples root

Rationale:

- avoids noisy explicit `baseUrl` in curated shared examples
- allows the same example to be staged under docs and served locally without rewriting source files

Proposed convention:

- data files under `examples/data/`
- reusable imported fragments under `examples/shared/`
- example specs use examples-root-relative URLs, for example:
  - `"url": "data/sincos.csv"`
  - `"import": { "url": "shared/genes.json" }`

Consequences:

- shared examples should not rely on spec-directory-relative paths
- shared examples should generally avoid relative imports like `../shared/foo.json`

This convention applies to curated shared examples under `examples/core/`, `examples/docs/`, and `examples/app/`. Docs-derived examples follow an additional “no imported views” rule described below.

## 6. Private projects should keep spec-directory-relative behavior

Rationale:

- private projects are effectively standalone visualization projects with colocated data
- developers should not need to reshape their project trees into the shared examples conventions

Consequences:

- opening `private/myProject/spec.json` should behave as if the spec lives in that folder
- the effective base should be the spec file directory

## 7. Website examples should get explicit `baseUrl` injected when opened in playground

Rationale:

- website examples already use spec-directory-relative assets and imports
- once opened in the playground, users may heavily edit them
- hidden inherited base state is confusing and stale once the spec diverges from the original

Consequences:

- when the playground opens a website-hosted example and the spec lacks `baseUrl`, it should inject `baseUrl` into the editor text
- after injection, the spec becomes self-contained and no hidden inherited base remains

## 8. Playground should still show the active base URL prominently

Rationale:

- transparency helps users understand why relative paths resolve the way they do
- useful for debugging and manual editing

Desired UI behavior:

- show current effective base URL
- indicate whether it is:
  - default examples root
  - explicit spec `baseUrl`
  - private project/spec source directory
  - injected website example `baseUrl`
- provide a clear button where applicable

## 9. Curated shared examples in the playground should behave consistently locally and when deployed

Rationale:

- a future dropdown or gallery should not need one behavior for local Vite development and another for the deployed playground
- environment-specific path hacks are the main source of current playground inconsistency
- local developer ergonomics still matter for direct app dev-server browsing

Consequences:

- the curated shared example browser should use the same catalog structure and loading behavior in both environments
- URLs do not need to be identical across environments if that would hurt developer ergonomics
- example catalog data should be generated once and consumed by both environments

## Target Repository Layout

```text
examples/
  core/
  app/
  data/
  shared/
  ...

private/
  my-project/
    spec.json
    data/
    ...
```

Notes:

- `examples/core/` and `examples/app/` are illustrative categories, not a strict requirement.
- Categories can be refined during migration.
- `examples/data/` and `examples/shared/` should be stable cross-example roots.
- repo-root `private/` is now the primary location even though legacy `packages/core/private/` is still supported during migration.

## Naming and Placement of Docs-Derived Examples

Docs-derived examples should live under `examples/docs/` and mirror the docs topic structure rather than using one flat directory.

Rationale:

- GenomeSpy docs already have a strong topical hierarchy.
- The path should tell a contributor where an example belongs without needing a lookup table.
- Hierarchical placement avoids long opaque flat filenames.
- It keeps examples close to the concepts they document, while still allowing reuse by path.

### Proposed convention

1. Mirror the docs path under `examples/docs/`.
2. Name each example after the nearest meaningful section heading in kebab-case.
3. Use a short qualifier when a page has multiple examples under one section.
4. Avoid generic names such as `interactive-example.json`.
5. Avoid repeating path context in the filename when the directory already provides it.
6. Include a succinct `description` and a `$schema` entry in each extracted docs example file.
7. Keep docs-derived examples simple and avoid imported views.

### Examples from the current docs

The following illustrate the intended naming style:

- `docs/index.md`
  - interactive overview example
  - proposed path: `examples/docs/index/interactive-overview.json`

- `docs/grammar/transform/coverage.md`
  - example showing both coverage and pileup
  - proposed path: `examples/docs/grammar/transform/coverage/coverage-pileup.json`

- `docs/grammar/composition/layer.md`
  - lollipop plot example
  - proposed path: `examples/docs/grammar/composition/layer/lollipop-plot.json`

- `docs/grammar/mark/point.md`
  - semantic zoom fraction example
  - proposed path: `examples/docs/grammar/mark/point/semantic-zoom-fraction.json`

### Filename heuristics

- Prefer concept names over UI wording.
  - Good: `coverage-pileup.json`
  - Avoid: `interactive-example.json`

- Prefer short descriptive names over full sentence headings.
  - Good: `semantic-zoom-fraction.json`
  - Avoid: `this-example-demonstrates-score-based-semantic-zoom.json`

- Do not repeat the surrounding directory context in the filename.
  - Good: `lollipop-plot.json`
  - Avoid: `layer-lollipop-plot-example.json`

- Use a qualifier only when needed to disambiguate multiple examples on the same page.
  - For example: `axis-basics.json` and `axis-dynamic-domain.json`

### Reused docs examples

If the same example is intentionally reused on multiple pages:

- keep one canonical file in the most natural topic folder
- reference that file from the other pages
- avoid duplicating the same spec under multiple docs paths

A shared docs-only folder may be added later if repeated reuse becomes common:

- `examples/docs/shared/`

That folder should be used sparingly. Most docs-derived examples should still live in the topic folder that best explains their purpose.

### Self-contained rule for docs examples

Docs-derived examples should stay simple and focused.

That means:

- prefer inline `values`, `sequence`, or similarly compact local spec data
- shared files under `examples/data/` are allowed when that keeps the spec readable
- when using shared data, use tidy examples-root-relative paths such as `data/sincos.csv`
- avoid dependencies on `examples/shared/`
- avoid imports from `examples/core/`
- avoid importing other specs or view fragments

Rationale:

- extracted docs examples should remain easy to understand in isolation
- docs examples should minimize coupling to the broader shared example corpus
- forbidding imported views keeps later reuse and maintenance simpler

### Required file metadata for extracted docs examples

Each extracted docs example should include:

- `"$schema": "https://cdn.jsdelivr.net/npm/@genome-spy/core/dist/schema.json"`
- a succinct top-level `description`

Rationale:

- `$schema` improves editor support and makes the example self-describing
- `description` gives catalog generation a stable human-readable summary

## Target Runtime Model

### Docs

- docs macro reads `examples/.../foo.json`
- macro inlines the exact JSON into the page
- macro emits a docs embed wrapper with explicit base URL for shared examples
- macro emits `Open in Playground`
- docs build stages `examples/` into the public docs tree under `/docs/examples/`

Docs-generated playground links may target the deployed-style staged path even in local docs builds. Local docs serving does not need special integration with a locally running playground.

### App dev server

- serves repo-root `examples/` at `/examples`
- serves repo-root `private/` at `/private`
- index page lists both sources separately

### Playground

The playground needs explicit source modes:

1. Shared monorepo examples
   - loaded from environment-appropriate curated example URLs
   - effective base defaults to the shared examples root
   - no injected `baseUrl` needed

2. Private projects
   - loaded from `/private/...`
   - effective base is the spec file directory

3. Website examples
   - loaded from `/examples/...` on `genome-spy.github.io`
   - inject explicit `baseUrl` into the spec if missing

### Playground parity between local and deployed environments

Local playground and deployed playground should use the same curated example catalog structure and the same source-mode logic.

The exact URL shape may differ by environment when that improves ergonomics:

- deployed docs links may use `/docs/examples/...`
- local app dev-server and local playground browsing may use `/examples/...`

Different source types may still use different paths:

- curated shared examples: environment-specific shared-example URLs
- private projects: `/private/...`
- website showcase examples: `/examples/...`

The key requirement is behavioral parity, not identical strings in the address bar.

### Example catalog for the playground

If the playground gains a dropdown or gallery for curated examples, it should be driven by one generated catalog.

Catalog update mechanism:

- local development should generate the catalog at runtime from the filesystem
- docs/playground builds should generate the same catalog shape at build time
- the catalog should not be checked in as a hand-maintained source file
- adding, renaming, or removing an example file should automatically affect the next local run or build without manual catalog edits

Suggested fields:

- `id`
- `title`
- `category`
- `specPath`
- `specUrl`
- `sourceMode`

For curated shared examples:

- `title` should come from the example `description`, with a filename-based fallback only when `description` is missing
- `specUrl` should resolve to the correct shared-example URL for the current environment
- `sourceMode` should be `shared-example`

The catalog should exclude:

- `private/`
- external `website-examples`

Private projects and website examples can still be opened by URL, but they should not be mixed into the curated shared example browser by default.

The curated catalog should include:

- `examples/core/`
- `examples/docs/`
- `examples/app/`

### Tests

- enumerate `examples/**/*.json`
- validate/parse them
- snapshot normalized outputs where appropriate
- exclude `private/`
- exclude `examples/app/` from automated testing for now
- do not attempt to validate external `website-examples` from this repository

## Base URL Rules

### Shared monorepo examples

Shared monorepo examples should rely on the examples root.

Examples:

- `data/sincos.csv`
- `shared/cytobands.json`

The runtime must set the effective base to the staged examples root:

- local app dev-server and local playground development: `/examples/`
- staged docs/public docs build: `/docs/examples/`

This is intentionally different from spec-directory-relative behavior.

### Private projects

Private specs use the spec file directory as base:

- `private/myProject/spec.json` resolves `data/foo.tsv` against `private/myProject/`

### Website examples

Website examples use the spec file directory as base, but the playground should materialize that by injecting `baseUrl` into the editor text on load.

Example:

- source URL: `/examples/OCAC/ocac.json`
- injected `baseUrl`: `/examples/OCAC/`

### Explicit spec `baseUrl`

If a spec already defines `baseUrl`, it wins.

## Docs Macro Design

Introduce a docs macro in the custom Markdown extension for file-backed examples.

Possible syntax:

```text
EXAMPLE examples/core/marks/point/basic.json height=200 spechidden=true
```

Responsibilities:

- read the source example JSON file
- inline the exact JSON contents into the rendered code block
- emit `<genome-spy-doc-embed ...>` with:
  - explicit base URL for the embedded viewer
  - any requested UI properties such as `height` and `spechidden`
- add a link below the spec:
  - `Open in Playground`

For shared monorepo examples, that link should point to the staged public docs asset, for example:

- `/playground/?spec=/docs/examples/core/marks/point/basic.json`

The macro should not support `private/`.

## Docs Asset Staging

Extend docs asset preparation to stage shared examples into the docs tree.

Desired staged path:

- source: `examples/...`
- staged docs path: `docs/examples/...`
- deployed public path: `/docs/examples/...`

This should be done in the existing docs preparation script so docs builds remain deterministic.

## Dev Server Changes

Update the app dev server to serve repo-root content instead of package-local content:

- `/examples` -> repo-root `examples/`
- `/private` -> repo-root `private/`

The index page should clearly separate:

- Shared examples
- Private projects

The listing logic should favor full project folders in `private/`, not just a flat list of files.

If legacy `packages/core/private/` still exists, the dev-server should:

- log a warning to the console
- show a warning in the generated index page

## Playground Changes

### Source tracking

The playground should persist structured state, not just spec text.

Suggested persisted shape:

```json
{
  "specText": "...",
  "sourceUrl": "https://genomespy.app/docs/examples/docs/grammar/transform/coverage/coverage-pileup.json",
  "sourceMode": "shared-example"
}
```

Possible `sourceMode` values:

- `default`
- `shared-example`
- `private-project`
- `website-example`
- `detached`

### URL handling

- resolve `?spec=` using `new URL(specParam, window.location.href)`
- stop using regex parsing for `baseUrl`
- stop clearing the query string after load
- preserve and restore source URLs that use the curated shared-example public path

### Effective base computation

Rules:

- explicit `spec.baseUrl` wins
- `shared-example` defaults to shared examples root
- `private-project` uses the source spec directory
- `website-example` injects `baseUrl` into the spec if missing

### Editor mutation behavior

For shared examples:

- keep specs clean by not auto-injecting `baseUrl`
- show active base source in the UI

For website examples:

- inject `baseUrl` immediately on load if absent
- mark the loaded editor state as detached immediately after injection
- this avoids hidden stale state after user edits

### UI additions

Add a visible section in the playground toolbar or nearby panel that shows:

- source URL, if any
- effective base URL
- source mode

Optional controls:

- clear explicit `baseUrl`
- reapply derived base URL
- detach from source

The first step only requires visibility and a clear button where applicable.

## Website Repo Changes

The website repo should stop linking into the playground using relative cross-app paths such as:

- `playground/?spec=../examples/OCAC/ocac.json`

Instead use absolute site-rooted paths:

- `/playground/?spec=/examples/OCAC/ocac.json`

Rationale:

- avoids dependence on current page path
- makes the source URL unambiguous
- aligns with the new playground source tracking model

## Screenshot Capture Follow-up

Status: partially completed on branch `examples-reorg`

This should be implemented as a browser-driven export workflow, not as a pure
Node-side renderer.

Rationale:

- GenomeSpy rendering is WebGL-backed, so screenshots should be captured from a
  real browser context
- the existing `exportCanvas()` API in core already provides a stable PNG export
  hook once a spec has been embedded and rendered
- a browser harness keeps screenshot generation close to real playground/docs
  behavior

### Output placement

Place each screenshot next to its spec in the curated examples tree.

Examples:

- `examples/core/layout/grid/complex_grid_layout.json`
- `examples/core/layout/grid/complex_grid_layout.png`
- `examples/docs/grammar/mark/rule/ranged-rules.json`
- `examples/docs/grammar/mark/rule/ranged-rules.png`

Rationale:

- screenshots remain discoverable next to the source spec
- renames and moves stay localized
- the future playground/docs catalog can resolve screenshot paths by convention
  instead of extra metadata

### Initial image format

Start with PNG.

Rationale:

- core already exports PNG via `exportCanvas()`
- no extra image conversion step is needed initially
- deterministic lossless output is preferable while the workflow is still
  settling

If storage size later becomes an issue, consider adding an optional post-process
step to derive WebP thumbnails for docs/playground use. Keep the authoritative
golden capture format lossless at first.

### Capture mechanism

Use Playwright to drive a dedicated screenshot harness page served from the
repository.

Proposed flow:

1. Start a local server that can serve curated examples at `/examples/...`.
2. Open a dedicated screenshot harness page in a real browser.
3. Pass the target spec URL to the harness via query string.
4. Wait until the embedded visualization reports that initial rendering is
   complete.
5. Export the visualization with `exportCanvas(...)`.
6. Decode the returned data URL and write the PNG next to the source spec.

The harness page should use the same spec-loading behavior as other curated
example entry points:

- curated examples resolve relative paths against `/examples/`
- no special-case `baseUrl` rewriting inside the screenshot script

### Harness location

Prefer a small harness page under `packages/core/` or another lightweight local
dev-server entry point rather than routing this through the full playground UI.

Rationale:

- fewer moving parts than the playground
- easier to control container size and export timing
- closer to the low-level rendering API that actually produces the PNG

The current `packages/core/dev-server.mjs` is a good starting point because it
already serves curated examples and can host an additional screenshot-only page.

### Sizing rules

Screenshots should reflect the visualization's intended canvas size where that
size is explicit, while still producing deterministic output for container-sized
specs.

Proposed rules:

1. If the spec resolves to an explicit finite width and height, export at that
   size.
2. If either dimension remains container-driven or otherwise unresolved, render
   the screenshot harness container at `600x400` logical pixels.
3. If one dimension is explicit and the other is container-driven, keep the
   explicit dimension and use the fallback size for the unresolved axis.

Examples:

- explicit `width: 800`, `height: 120` -> export `800x120`
- `width: "container"`, `height: 130` -> export `600x130`
- no explicit dimensions -> export `600x400`

This rule should be implemented in terms of the actual resolved canvas size
after embedding, not by guessing directly from raw JSON fields. The screenshot
harness should inspect the embedded view's logical canvas size and only fall
back for unresolved dimensions.

### Device pixel ratio

Start with a fixed device pixel ratio of `1` for deterministic output.

Rationale:

- avoids machine-dependent Retina/non-Retina differences
- keeps file sizes moderate
- enough for catalog thumbnails and regression review

If higher-resolution exports are later needed for marketing/docs assets, that
should be a separate explicit mode.

### Catalog integration

The curated example catalog should not store screenshot metadata manually.

Instead:

- the catalog generator can look for a sibling `.png` next to each `.json`
- when present, expose it as the example's screenshot URL
- when absent, the example still remains valid and listed

This keeps screenshots optional during rollout.

### Rollout strategy

Recommended order:

1. Add the screenshot harness page and local generation script.
2. Support single-example capture by explicit spec path.
3. Support batch generation for `examples/core/**` and `examples/docs/**`.
4. Teach the example catalog to surface sibling screenshots when available.
5. Later, discuss screenshot approval/update workflow and any CI integration.

### Draft commit messages

- `feat(core): add example screenshot harness`
- `feat(core): add curated example screenshot generator`
- `feat(playground): surface example screenshots in catalog`

Implementation notes:

- A local screenshot harness now exists at `packages/core/screenshot.html`.
- The harness is driven by `packages/core/src/screenshotHarness.js` and uses
  `embed(...).exportCanvas(...)` for PNG export.
- `packages/core/dev-server.mjs` now exposes `/__health` for script readiness
  checks and serves `/screenshot.html` directly instead of relying on Vite's
  HTML middleware path for liveness detection.
- The core embed API now exposes `getLogicalCanvasSize()` so the harness can use
  the resolved canvas dimensions when exporting.
- `packages/core/scripts/captureScreenshots.mjs` can capture a specific curated
  example or batch all curated `examples/core/**` and `examples/docs/**`
  examples, writing sibling `.png` files next to the source specs.
- The batch script currently excludes the same app-only and remote lazy-data
  examples that are unsuitable for the current offline structural test harness.
- The script expects the `playwright` package to be installed in the workspace.
  That package is not yet declared or installed in this branch, so the script
  currently fails fast with an explicit instruction instead of attempting a
  partial capture.
- The curated example catalog generator now looks for sibling `.png` files and
  exposes screenshot URLs when they exist.

## Risks and Mitigations

### Risk: mixed path semantics

Curated shared examples use examples-root-relative paths, while private projects and website examples use spec-directory-relative paths.

Mitigation:

- make the playground source mode explicit
- document the distinction in contributor docs
- keep docs macro limited to shared examples only

### Risk: docs/public path mismatches

Deployed docs links use `/docs/examples/...`, while local app and playground browsing prefer `/examples/...`.

Mitigation:

- centralize staged-path generation in the docs macro and asset prep
- generate environment-appropriate shared example URLs from one catalog shape
- avoid hardcoded assumptions scattered through the codebase

### Risk: stale hidden base URLs in playground

Mitigation:

- inject `baseUrl` for website examples
- show current effective base URL in the UI
- persist source metadata explicitly

### Risk: broad migration churn

Mitigation:

- split into phased commits
- keep mechanical path rewrites separate from behavioral changes
- migrate docs in batches

## Open Questions

- Do shared examples need a generated manifest immediately, or can initial docs/playground indexing use direct filesystem enumeration?
- Should shared example categories be `core/`, `app/`, and `docs/`, or should categories reflect topic areas such as `marks/`, `transforms/`, and `sample-collections/`?
- Do any current shared examples rely on spec-directory-relative imports in a way that makes the examples-root convention awkward?
- Should the playground expose an explicit "Insert current effective baseUrl into spec" action for shared examples too, or only for website examples?
